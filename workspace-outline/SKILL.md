# Workspace Outline Skill

本技能用于**协助用户**为工作区（Git 仓库、项目文件夹等）创建和维护**概念目录文件**。这些文件按功能/概念组织（而非物理目录结构），包含关键内容摘要、概念间引用、物理位置映射，帮助 LLM 在主动探索工作区时有迹可循，避免盲目探索导致上下文超限和处理时间过长。

## 核心理念

- **概念组织**：目录文件按概念/功能逻辑组织，不按物理目录结构复制
- **用户主导**：概念分类由用户决定，LLM 协助执行
- **增量构建**：每次更新只修改相关文件，保持累积效应
- **交叉引用**：概念之间建立丰富的引用关系

## 强制性约束

### 1. 入口文件必须有 YAML front matter

入口文件（`WORKSPACE_OUTLINE.md`）必须包含以下 YAML front matter：

```yaml
---
version: "1.0"           # 大纲格式版本
updated: "2026-04-10"    # 最后更新日期
description: "工作区描述"   # 一句话描述工作区是做什么的

# 可选：配置忽略模式（语法类似.gitignore）
config:
  ignore:
    - "*.pyc"
    - "__pycache__/"
    - ".git/"
---
```

**说明**：
- `version`: 大纲文件的格式版本号
- `updated`: 最后更新日期
- `description`: 工作区描述，让 LLM 快速了解工作区的整体用途
- `config.ignore` (可选): 指定 LLM 应该忽略的文件或目录模式

### 2. 文件路径使用相对路径

所有文件路径必须是相对于工作区根目录的相对路径：
- 正确：`outlines/authentication.md`
- 错误：`/Users/xxx/project/outlines/authentication.md`

### 3. 目录文件使用 Markdown 格式

所有目录文件使用 `.md` 扩展名。

### 4. 文件大小限制

- 每个文件不应超过 **3000 字符**
- 如超限，应拆分为多个概念文件

### 5. 交互确认（强制）

**在创建/更新任何目录文件之前，LLM 必须与用户确认**：

1. **识别阶段**：LLM 识别用户提到的概念
2. **确认阶段**：LLM 询问用户是否需要为该概念创建/更新目录文件
   - 示例："我注意到您提到了'鉴权模块'，需要我为它创建概念文件吗？"
3. **等待确认**：等待用户确认后再执行创建/更新
4. **验证阶段**：创建后，请用户确认内容是否正确

**原因**：概念分类由用户主导，避免 LLM 自主创建错误的概念结构。

## 目录结构

入口文件位于工作区根目录，所有概念文件存放在 `outlines/` 目录下（**不按物理结构组织**）：

```
workspace-root/
├── WORKSPACE_OUTLINE.md          # 入口文件：全局概念地图
└── outlines/                     # 概念文件（按功能/概念组织）
    ├── authentication.md         # 概念：鉴权
    ├── document-search.md        # 概念：文档搜索
    ├── user-session.md           # 概念：用户会话
    └── project-overview.md       # 纯描述：项目概述
```

## 入口文件格式

入口文件（WORKSPACE_OUTLINE.md）采用**索引 + 整体总结**的结构：

```yaml
---
version: "1.0"
updated: "2026-04-10"
description: "项目概念地图"
---

# 项目概念地图

## 工作区整体总结

用一段话或几段话描述整个工作区的整体情况，包括：
- 工作区的主要用途和目标
- 整体架构和设计思路
- 主要技术栈或框架
- 关键特性或亮点

| 概念 | 物理位置 |

## 核心概念关系图

用简单的 ASCII 图或文字描述本概念文件内部各子概念之间的关系。

示例：
```
[Session 数据模型] ← 包含 → [TokenUsageData]
       ↓
[会话存储] ← 存储 → [Session 数据模型]
       ↓
[会话管理器] ← 操作 → [会话存储]
```

## 核心内容
- 关键点 1：描述
- 关键点 2：描述
- ...

## 内部概念关联

用自然语言段落描述本概念文件内部各子概念之间的关系。

示例：
```
Session 数据模型是核心，TokenUsageData 作为其字段嵌入其中。

会话存储使用字典结构存储 Session 对象，以 session_id 为键实现快速查找。

会话管理器提供 API 操作会话存储，包括创建、获取、结束和清理会话。

会话生命周期描述了 Session 对象从创建到清理的状态变迁：
- 创建时生成 session_id 和 created_at
- 每次交互更新 last_activity
- 基于 last_activity 计算空闲时间判断过期
- 清理时从存储中移除 Session 对象
```

## 相关概念

## 原文引用

```

## 工作流程

### 场景 1：首次创建

当项目没有概念目录文件时：

1. **询问用户**："我注意到项目还没有概念目录，需要我帮您创建入口文件吗？"
2. 用户确认后：
   - 创建 `WORKSPACE_OUTLINE.md`（含 YAML front matter）
   - 创建 `outlines/` 目录
3. **请用户验证**

### 场景 2：创建新概念

当用户要创建一个新概念时：

1. **读取入口文件**：检查是否有相似概念（避免冲突）
2. **概念冲突检查**：
   - 如发现潜在冲突，向用户报告：
     ```
     我发现 [鉴权模块] 可能与您要创建的 [用户认证] 概念重叠
     - 鉴权模块：负责用户身份验证和授权
     - 您想创建的用户认证：[用户描述]
     
     请问您想要：
     a) 更新现有的 [鉴权模块]
     b) 创建新的 [用户认证]，并建立引用关系
     c) 合并两个概念
     ```
3. **询问用户确认**："需要我为 [概念名] 创建概念文件吗？"
4. 用户确认后：
   - 创建概念文件（`outlines/concept-name.md`）
   - 在入口文件中添加该概念
5. **请用户验证**

### 场景 3：更新现有概念

当用户要更新一个现有概念时：

1. **读取入口文件**：定位到相关概念文件
2. **读取概念文件**：获取现有内容
3. **询问用户确认**："我可以更新 [概念名]，添加关于 [内容] 的信息，可以吗？"
4. 用户确认后，更新概念文件
5. **请用户验证**

### 场景 4：概念合并

当用户要合并两个概念时：

1. **分析两个概念的内容**：
   - 哪些内容保留
   - 哪些内容重复
   - 哪些引用需要更新
2. **向用户报告合并计划**：
   ```
   合并 [A] 和 [B] 为 [C]：
   - 保留：A 的摘要、B 的核心内容
   - 删除：重复的描述
   - 更新：引用 [A] 或 [B] 的其他页面 → [C]
   确认执行？
   ```
3. 用户确认后执行合并
4. **请用户验证**

## 避免全量扫描

**问题**：每次更新都读取所有文件会消耗大量 token 和时间。

**解决方案**：

1. **入口文件作为索引**：
   - LLM 只读取入口文件（作为索引）
   - 从入口文件中定位相关概念
   - 只读取相关概念文件进行更新

2. **物理位置反向映射**：
   - 入口文件中记录每个概念的关键物理位置
   - 当用户提到某个文件时，LLM 可以快速找到所属概念

## 不要做的事（重要）

❌ **不要按物理目录结构复制** - 概念文件按功能/逻辑组织
❌ **不要自主决定概念分类** - 必须先询问用户
❌ **不要一次性创建多个概念文件** - 一次只处理一个概念
❌ **不要创建超过 3000 字符的文件** - 超限应拆分
❌ **不要忽略概念冲突检查** - 创建前先检查入口文件

## 询问用户模板

**创建概念前：**
```
我注意到您提到了 [概念名]，需要我为它创建概念文件吗？
```

**概念冲突时：**
```
我发现 [现有概念] 可能与您要创建的 [新概念] 概念重叠
- 现有概念：[摘要]
- 新概念：[用户描述]

请问您想要：
a) 更新现有的 [现有概念]
b) 创建新的 [新概念]，并建立引用关系
c) 合并两个概念
```

**更新概念前：**
```
我可以更新 [概念名]，添加关于 [内容] 的信息，可以吗？
```

**验证时：**
```
我已创建/更新了 [概念名]，请确认内容是否正确。
```

## 用户手动触发的维护操作

以下操作由用户主动触发，LLM 不自动执行：

### 1. Lint 检查

用户指令："请检查概念目录是否有孤立概念或过时内容"

LLM 执行：
- 扫描所有概念页面
- 找出没有 inbound links 的概念
- 找出没有 outbound links 的概念
- 报告并建议建立引用

### 2. 概念合并

用户指令："请合并 [A] 和 [B] 两个概念"

LLM 执行：
- 分析两个概念的内容
- 报告合并计划
- 用户确认后执行

### 3. 验证物理位置

用户指令："请验证概念目录中的物理位置映射是否正确"

LLM 执行：
- 检查物理位置映射中的文件是否存在
- 对比文件内容与概念描述
- 报告潜在的不一致

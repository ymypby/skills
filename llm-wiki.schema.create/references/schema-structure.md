# Schema 结构定义

## Schema 文件

### 文件命名规范

- 使用 kebab-case 格式（全小写，单词间用连字符 `-` 连接）
- 文件名应简洁反映 schema 的核心领域，如 `testing-procedure`、`api-design`
- 避免使用缩写或含义不清的词汇

### Front Matter 格式

```yaml
---
description: >
  适用的领域、场景、提取目标。
  领域背景、适用场景、主要提取方向。
created: "YYYY-MM-DD"
---
```

### 正文内容结构

Schema 正文应包含以下结构要素（按需选用）：

| 结构要素 | 说明 | 示例 |
|----------|------|------|
| 关注范围 | 明确提取时应聚焦的内容领域和上下文 | user-story、测试流程 |
| 提取规则 | 具体的知识提取约束和偏好 | 保持步骤完整性 |
| 排除范围 | 明确不应提取的内容类型 | 根因分析、架构设计 |
| 页面类型偏好 | 明确的页面类型约束和偏好 | 首选类型、可选类型、禁止类型、类型判断依据 |
| 特殊约定 | 针对该领域的特定规则或格式要求 | 特定字段格式 |

### 示例

```yaml
---
description: >
  适用于软件测试领域的知识提取。
  主要从问题修复对话、需求讨论、功能描述中提取测试相关的知识。
created: "2026-04-24"
---

## 关注范围

关注 user-story 和对应的测试流程。

## 提取规则

- 提取测试步骤时保持完整性和可执行性
- 保留测试前置条件和预期结果

## 排除范围

排除问题根因分析、代码实现细节、系统架构设计。

## 页面类型偏好

Schema 必须明确指定页面类型偏好，以约束知识提取时的页面创建行为。

### 格式要求

| 子项 | 说明 | 必填 |
|------|------|------|
| 首选类型 | 此领域优先创建的页面类型 | 是 |
| 可选类型 | 在特定条件下可创建的页面类型 | 否 |
| 禁止类型 | 此领域不应创建的页面类型 | 否 |
| 类型判断依据 | 帮助模型判断何时创建何种类型的规则 | 是 |

### 可选的 llm-wiki 页面类型

| 类型 | 说明 | 适用场景 |
|------|------|----------|
| `overview` | 领域总体视图，导航入口 | 高层次全局概览，整合多个概念和实体 |
| `entity` | 领域实体，独立事物 | 可识别、可命名的独立事物（不限于代码相关） |
| `procedure` | 流程/操作，可执行步骤 | "通过做什么完成了什么"，关注"怎么做" |
| `concept` | 抽象理念/模式 | 抽象的理念、模式、原则、设计思想 |
| `synthesis` | 综合分析 | 对多个源文件/概念的深度分析、对比报告 |
| `source` | 源文件摘要 | 大文件或重要非代码文档的摘要 |

### 示例

```markdown
## 页面类型偏好

### 首选类型
- `procedure`：提取测试流程时创建

### 可选类型
- `entity`：当涉及新的测试对象（如测试框架、测试工具）时创建

### 禁止类型
- `synthesis`：不创建综合分析页面
- `source`：不创建源文件摘要页面

### 类型判断依据
- 包含可执行步骤的内容 → procedure
- 描述独立可识别事物 → entity
- 描述抽象理念或设计模式 → concept
```
```

## Schema Index

### 格式规范

```markdown
# Schema Index

- **schema-name**
  - 描述：description 内容摘要。
  - 路径：`schema-name.md`
```

### 示例

```markdown
# Schema Index

- **testing-procedure**
  - 描述：适用于软件测试领域的知识提取。主要从问题修复对话、需求讨论、功能描述中提取测试相关的知识。
  - 路径：`testing-procedure.md`

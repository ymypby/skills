# Lint 工作流指南

本文档详细说明 LLM Wiki 的 Lint（维护健康）操作流程，采用**三阶段循环交互模型**。

## 三阶段循环交互模型

### 阶段一：检查问题（只读模式）
### 阶段二：用户确认（等待模式）
### 阶段三：执行修复（写入模式）

---

## 阶段一：检查问题（只读模式）

**此阶段只读取文件，禁止任何修改操作。**

### 输入方式

用户可以提供以下任意输入：
- **单个页面路径**：如 `wiki/overview/project-overview.md`
- **多个页面路径**：如 `wiki/overview/project-overview.md wiki/procedures/dev-server-lifecycle.md`
- **目录路径**：如 `wiki/`（递归扫描所有 .md 文件）

### 步骤 1: 收集页面信息

读取用户指定范围内的所有 wiki 页面，提取每个页面的：
- `path`: 页面相对路径
- `relations`: 关联关系数组（包含 `path` 和 `desc`）

### 步骤 2: 验证路径有效性

```
对于每个页面 P：
  对于 P.relations 中的每个关系 R：
    1. 验证 R.path 路径是否有效（文件是否存在）
    2. 如果文件不存在 → 标记为"无效路径"问题
```

**问题类型**：
- **无效路径**：R.path 指向的文件不存在

### 步骤 3: 验证 desc 格式

对于每个关系条目，检查 desc 是否：
- **长度合规**：不超过 50 字
- **描述清晰**：能准确表达两个页面之间的关系

### 步骤 4: 关系正确性验证（语义验证）

**验证方式**：LLM 直接读取 wiki 页面正文内容，理解其含义后判断关系是否正确。

#### 步骤 4.1: 验证 relations 关系的正确性

```
对于每个页面 A 的每个 relations 条目（目标页面 B，desc: "xxx"）：
  1. 读取 A 的完整正文内容（排除 front matter）
  2. 读取 B 的完整正文内容（排除 front matter）
  3. LLM 理解 A 和 B 的正文内容，判断：
     - A 的正文中是否提及了 B（直接引用或内容相关）
     - B 的正文中是否提及了 A（直接引用或内容相关）
     - A 和 B 的内容是否存在关联关系
  4. 综合判断结果：
     - 关系正确 ✓：A 或 B 的正文确实提及对方，或内容确实相关，且 desc 准确
     - 关系误标 ✗：A 和 B 的正文没有任何关联 → 建议"删除"
     - 描述不准确 ✗：关系存在但 desc 与实际关联不符 → 建议"更新"
```

#### 步骤 4.2: 检测关系遗漏

```
对于每个页面 P：
  1. 读取 P 的完整正文内容
  2. LLM 理解 P 的正文，判断 P 是否提及了其他 wiki 页面
  3. 对比 P 的 relations 声明，检查是否有遗漏：
     - 如果 P 正文中提及了页面 X，但 relations 中没有声明 → 建议"新增"关系
```

**问题类型**：
- **关系误标**：front matter 中声明了关系，但正文中不存在任何关联 → 建议"删除"
- **关系遗漏**：正文中实际提及了其他页面但未声明 → 建议"新增"
- **描述不准确**：关系存在但 desc 与正文实际内容不符 → 建议"更新"

### 步骤 5: 检测孤立页面

- **无关联**：`relations` 为空的页面（且不是 index.md 和 log.md）
- 报告孤立页面，建议建立链接

### 步骤 6: 检查矛盾/过时内容

- 扫描所有页面，查找相互矛盾的描述
- 检查是否有页面描述与当前代码不一致
- 报告发现的问题

---

## 阶段二：用户确认（等待模式）

**此阶段必须等待用户明确确认，禁止自动执行修复。**

### 生成 Lint 报告

向用户展示检查结果：

```
Lint 报告：
==========

【路径验证问题】
- 无效路径: X 个
  - 页面A 的 relations 包含路径 xxx，但该路径不存在

【关系正确性问题】
- 关系误标: M 个
  - 页面A 声明与页面B 有关系，但正文中不存在任何关联
- 关系遗漏: N 个
  - 页面A 正文中提及了页面B，但 relations 中未声明
- 描述不准确: O 个
  - 页面A 和页面B 的关系 desc 与正文实际内容不符

【孤立页面】
- 无关联: [列表]

【内容问题】
- 矛盾内容: [列表] 或 "未发现矛盾内容 ✓"

【修复计划】
1. 删除无效路径: X 处
2. 删除误标关系: M 处
3. 新增遗漏关系: N 处
4. 更新不准确描述: O 处

请选择后续操作：
- "全部修复"：执行所有建议的修复操作
- "部分修复"：指定需要修复的项目
- "跳过"：仅报告问题，不执行任何修改
- "通过"/"没问题"：确认当前状态，结束循环
```

### 等待用户响应

- **全部修复**：执行所有建议的修复操作，进入阶段三
- **部分修复**：用户指定需要修复的项目，进入阶段三
- **跳过**：仅报告问题，不执行任何修改，自动回到阶段一重新检查
- **通过/没问题**：确认当前状态，终止循环

---

## 阶段三：执行修复（写入模式）

**仅在用户确认后执行。**

### 修复步骤

1. **删除无效路径**：
   - 从 relations 中移除路径不存在的关系条目
   - 使用 `replace_in_file` 更新 front matter

2. **删除误标关系**：
   - 从 relations 中移除正文中不存在关联的关系条目

3. **新增遗漏关系**：
   - 根据正文中的实际提及关系，在 relations 中添加对应条目
   - desc 需准确描述两页面间的关联

4. **更新不准确描述**：
   - 根据正文中的实际内容，更新 desc 使其准确反映关联关系

5. **更新 index.md**：
   - 同步更新 index.md 中对应页面的关系信息

6. **追加 log.md**：
   - 按照格式追加 lint 记录：
   ```markdown
   ## [YYYY-MM-DD] lint
   
   - 扫描: N 个页面
   - 修复: M 个关系问题（无效路径: X, 关系误标: Y, 关系遗漏: Z, 描述不准确: W）
   - 发现: K 个孤立页面
   ```

### 输出修复结果

```
修复完成：
==========
- 已删除 X 个无效路径
- 已删除 Y 个误标关系
- 已新增 Z 个遗漏关系
- 已更新 W 处不准确描述
- 已更新 index.md 和 log.md

现在重新执行阶段一，检查修复结果...
```

---

## 循环机制

**阶段三完成后或阶段二用户选择"跳过"后，自动回到阶段一重新检查。**

### 循环流程

1. 阶段一检查完成后，进入阶段二展示报告
2. 阶段二用户响应：
   - 选择"全部修复"/"部分修复" → 进入阶段三执行修复 → 修复完成后自动回到阶段一
   - 选择"跳过" → 自动回到阶段一重新检查
   - 选择"通过"/"没问题" → 终止循环
3. 循环持续直到用户在阶段二明确确认通过

### 循环终止

循环在以下情况下终止：
- 用户在阶段二明确反馈"通过"、"没问题"、"好了"等确认词

---

## Lint 触发时机

- 用户主动指令："请检查 wiki 健康"
- 用户主动指令："请更新 relations"
- 用户提供页面路径或目录路径进行验证
- 大型 Ingest 完成后（可选，先询问用户）

---

## 工作流状态追踪

每个操作完成后，在 log.md 中追加记录：

```markdown
## [YYYY-MM-DD] lint

- 扫描: N 个页面
- 修复: M 个关系问题
- 发现: K 个孤立页面
```

## 注意事项

- **路径有效性必须验证**：relations 中的每个 path 必须指向存在的文件
- **严格三阶段循环**：检查、确认、修复按顺序执行，禁止跳过确认环节
- **阶段一只读**：检查阶段禁止修改任何文件
- **阶段二等待**：必须等待用户明确确认
- **阶段三写入**：仅在确认后执行修复
- **循环机制**：阶段三修复完成后或阶段二用户选择"跳过"后，自动回到阶段一重新检查
- **关系正确性必须验证**：通过阅读原文理解语义，验证声明的关系是否真实存在
- **index.md 保持一致**：每次关系变更后同步更新索引
- **无效路径必须报告**：引用不存在的页面是严重问题

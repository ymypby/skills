# Lint 工作流指南

本文档详细说明 LLM Wiki 的 Lint（维护健康）操作流程，采用**三阶段循环交互模型**。

## 三阶段循环交互模型

### 阶段一：检查问题（只读模式）
### 阶段二：用户确认（等待模式）
### 阶段三：执行修复（写入模式）

---

## 阶段一：检查问题（只读模式）

**此阶段只读取文件，禁止任何修改操作。**

### 输入方式

用户可以提供以下任意输入：
- **单个页面路径**：如 `wiki/overview/project-overview.md`
- **多个页面路径**：如 `wiki/overview/project-overview.md wiki/procedures/dev-server-lifecycle.md`
- **目录路径**：如 `wiki/`（递归扫描所有 .md 文件）

### 步骤 1: 收集页面信息

读取用户指定范围内的所有 wiki 页面，提取每个页面的：
- `path`: 页面相对路径
- `references`: 引用关系数组
- `referenced_by`: 被引用关系数组

### 步骤 2: 正向验证（references → referenced_by）

```
对于每个页面 P：
  对于 P.references 中的每个关系 R：
    1. 验证 R.page 路径是否有效（文件是否存在）
    2. 读取 R.page 指向的目标页面 T
    3. 验证 T.referenced_by 中是否存在页面字段等于 P.path 的条目
       - 如果不存在 → 标记为"缺失反向引用"问题
       - 如果存在但 description 不匹配 → 标记为"描述不一致"问题
```

**问题类型**：
- **缺失反向引用**：P 引用了 T，但 T 的 referenced_by 没有 P
- **描述不一致**：P.references 中的 description 与 T.referenced_by 中对应条目的 description 不同
- **无效路径**：R.page 指向的文件不存在

### 步骤 3: 反向验证（referenced_by → references）

```
对于每个页面 T：
  对于 T.referenced_by 中的每个关系 B：
    1. 验证 B.page 路径是否有效（文件是否存在）
    2. 读取 B.page 指向的源页面 P
    3. 验证 P.references 中是否存在页面字段等于 T.path 的条目
       - 如果不存在 → 标记为"幽灵引用"问题
       - 如果存在但 description 不匹配 → 标记为"描述不一致"问题
```

**问题类型**：
- **幽灵引用**：T 的 referenced_by 包含 P，但 P 没有引用 T
- **描述不一致**：T.referenced_by 中的 description 与 P.references 中对应条目的 description 不同
- **无效路径**：B.page 指向的文件不存在

### 步骤 4: 验证 description 格式

对于每个关系条目，检查 description 是否：
- **长度合规**：不超过 50 字
- **描述清晰**：能准确表达两个页面之间的关系类型
- **关系类型正确**：
  - references 的 description 应描述"我引用它的原因"
  - referenced_by 的 description 应描述"它引用我的原因"

### 步骤 5: 关系正确性验证（语义验证）

**验证方式**：LLM 直接读取 wiki 页面正文内容，理解其含义后判断关系是否正确。

#### 步骤 5.1: 验证 references 关系的正确性

```
对于每个页面 A 的每个 references 条目（目标页面 B，description: "xxx"）：
  1. 读取 A 的完整正文内容（排除 front matter）
  2. 读取 B 的完整正文内容（排除 front matter）
  3. LLM 理解 A 和 B 的正文内容，判断：
     - A 的正文中是否真的引用了 B 来描述、解释某些内容
     - B 的内容是否确实是 A 所描述的关系
  4. 综合判断结果：
     - 关系正确 ✓：A 确实引用了 B，且 description 准确
     - 关系误标 ✗：A 的正文中没有引用 B 的内容 → 建议"删除"
     - 描述不准确 ✗：关系存在但 description 与正文不符 → 建议"更新"
```

#### 步骤 5.2: 验证 referenced_by 关系的正确性

```
对于每个页面 B 的每个 referenced_by 条目（源页面 A，description: "xxx"）：
  1. 读取 A 的完整正文内容（排除 front matter）
  2. 读取 B 的完整正文内容（排除 front matter）
  3. LLM 理解 A 和 B 的正文内容，判断：
     - A 的正文中是否真的引用了 B
     - B 的 referenced_by 中描述的关系是否真实存在
  4. 综合判断结果：
     - 关系正确 ✓：关系存在且 description 准确
     - 关系误标 ✗：A 的正文中没有引用 B → 建议"删除"
     - 描述不准确 ✗：关系存在但 description 与正文不符 → 建议"更新"
```

#### 步骤 5.3: 检测关系遗漏

```
对于每个页面 P：
  1. 读取 P 的完整正文内容
  2. LLM 理解 P 的正文，判断 P 是否引用了其他 wiki 页面
  3. 对比 P 的 references 声明，检查是否有遗漏：
     - 如果 P 正文中引用了页面 X，但 references 中没有声明 → 建议"新增"关系
  4. 同样的逻辑检查 referenced_by 的遗漏
```

**问题类型**：
- **关系误标**：front matter 中声明了关系，但正文中不存在 → 建议"删除"
- **关系遗漏**：正文中实际引用了其他页面但未声明 → 建议"新增"
- **描述不准确**：关系存在但 description 与正文实际内容不符 → 建议"更新"

### 步骤 6: 检测孤立页面

- **无入链**：`referenced_by` 为空的页面（且不是 index.md 和 log.md）
- **无出链**：`references` 为空的页面（source 类型页面除外）
- 报告孤立页面，建议建立链接

### 步骤 7: 检查矛盾/过时内容

- 扫描所有页面，查找相互矛盾的描述
- 检查是否有页面描述与当前代码不一致
- 报告发现的问题

---

## 阶段二：用户确认（等待模式）

**此阶段必须等待用户明确确认，禁止自动执行修复。**

### 生成 Lint 报告

向用户展示检查结果：

```
Lint 报告：
==========

【结构验证问题】
- 缺失反向引用: X 个
  - 页面A → 页面B，但页面B的referenced_by缺少页面A
- 幽灵引用: Y 个
  - 页面B的referenced_by包含页面A，但页面A没有引用页面B
- 描述不一致: Z 个
  - 页面A和页面B之间的关系描述存在差异
- 无效路径: W 个
  - 页面A引用的路径xxx不存在

【关系正确性问题】
- 关系误标: M 个
  - 页面A声明引用页面B，但正文中不存在该引用关系
- 关系遗漏: N 个
  - 页面A正文中引用了页面B，但front matter中未声明
- 描述不准确: O 个
  - 页面A和页面B的关系description与正文实际内容不符

【孤立页面】
- 无入链: [列表]
- 无出链: [列表]

【内容问题】
- 矛盾内容: [列表] 或 "未发现矛盾内容 ✓"

【修复计划】
1. 修复缺失反向引用: X 处
2. 修复幽灵引用: Y 处
3. 统一描述不一致: Z 处
4. 删除误标关系: M 处
5. 新增遗漏关系: N 处
6. 更新不准确描述: O 处

请选择后续操作：
- "全部修复"：执行所有建议的修复操作
- "部分修复"：指定需要修复的项目
- "跳过"：仅报告问题，不执行任何修改
- "通过"/"没问题"：确认当前状态，结束循环
```

### 等待用户响应

- **全部修复**：执行所有建议的修复操作，进入阶段三
- **部分修复**：用户指定需要修复的项目，进入阶段三
- **跳过**：仅报告问题，不执行任何修改，自动回到阶段一重新检查
- **通过/没问题**：确认当前状态，终止循环

---

## 阶段三：执行修复（写入模式）

**仅在用户确认后执行。**

### 修复步骤

1. **修复缺失反向引用**：
   - 对于每个缺失的反向引用，在目标页面的 referenced_by 中添加对应条目
   - 使用 `replace_in_file` 更新 front matter

2. **修复幽灵引用**：
   - 从 referenced_by 中移除不存在对应 references 的条目

3. **修复描述不一致**：
   - 统一 references 和 referenced_by 中的 description
   - 优先保留更详细的描述

4. **删除误标关系**：
   - 从 references 或 referenced_by 中移除正文中不存在的关系条目

5. **新增遗漏关系**：
   - 根据正文中的实际引用关系，在 references 中添加对应条目
   - 同时在目标页面的 referenced_by 中添加反向条目

6. **更新不准确描述**：
   - 根据正文中的实际内容，更新 description 使其准确反映引用关系

7. **更新 index.md**：
   - 同步更新 index.md 中对应页面的引用关系

8. **追加 log.md**：
   - 按照格式追加 lint 记录：
   ```markdown
   ## [YYYY-MM-DD] lint
   
   - 扫描: N 个页面
   - 修复: M 个关系问题（缺失反向引用: X, 幽灵引用: Y, 描述不一致: Z, 关系误标: A, 关系遗漏: B, 描述不准确: C）
   - 发现: K 个孤立页面
   ```

### 输出修复结果

```
修复完成：
==========
- 已更新 X 个页面的 referenced_by
- 已修复 Y 个幽灵引用
- 已统一 Z 处描述不一致
- 已删除 A 个误标关系
- 已新增 B 个遗漏关系
- 已更新 C 处不准确描述
- 已更新 index.md 和 log.md

现在重新执行阶段一，检查修复结果...
```

---

## 循环机制

**阶段三完成后或阶段二用户选择"跳过"后，自动回到阶段一重新检查。**

### 循环流程

1. 阶段一检查完成后，进入阶段二展示报告
2. 阶段二用户响应：
   - 选择"全部修复"/"部分修复" → 进入阶段三执行修复 → 修复完成后自动回到阶段一
   - 选择"跳过" → 自动回到阶段一重新检查
   - 选择"通过"/"没问题" → 终止循环
3. 循环持续直到用户在阶段二明确确认通过

### 循环终止

循环在以下情况下终止：
- 用户在阶段二明确反馈"通过"、"没问题"、"好了"等确认词

---

## Lint 触发时机

- 用户主动指令："请检查 wiki 健康"
- 用户主动指令："请更新 relations"
- 用户提供页面路径或目录路径进行验证
- 大型 Ingest 完成后（可选，先询问用户）

---

## 工作流状态追踪

每个操作完成后，在 log.md 中追加记录：

```markdown
## [YYYY-MM-DD] lint

- 扫描: N 个页面
- 修复: M 个关系问题
- 发现: K 个孤立页面
```

## 注意事项

- **双向验证同等重要**：references 和 referenced_by 都必须保持同步
- **严格三阶段循环**：检查、确认、修复按顺序执行，禁止跳过确认环节
- **阶段一只读**：检查阶段禁止修改任何文件
- **阶段二等待**：必须等待用户明确确认
- **阶段三写入**：仅在确认后执行修复
- **循环机制**：阶段三修复完成后或阶段二用户选择"跳过"后，自动回到阶段一重新检查
- **关系正确性必须验证**：通过阅读原文理解语义，验证声明的关系是否真实存在
- **index.md 保持一致**：每次关系变更后同步更新索引
- **无效路径必须报告**：引用不存在的页面是严重问题

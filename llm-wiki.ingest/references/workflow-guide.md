# Ingest 工作流指南

本文档详细说明 LLM Wiki 的 Ingest（摄取源内容）操作流程。

## Ingest（摄取源内容）

当用户提供文件/代码或要求在对话上下文中提取知识时，执行 Ingest 流程。

**重要：Ingest 必须遵循三阶段确认流程（规划→确认→执行）。LLM 不得在未确认的情况下自动创建任何 wiki 页面。**

### 知识来源

| 来源类型 | 说明 | 示例 |
|----------|------|------|
| **文件/代码** | Git 仓库中的源文件、PDF、Word 文档等 | `ARCHITECTURE.md`, `src/agent.py`, `data/report.pdf` |
| **当前对话上下文** | 用户与 LLM 的对话中讨论的概念、决策、发现 | 用户描述系统设计、讨论技术方案、分享业务理解 |

### 三阶段流程概览

```
┌─────────────────────────────────────────┐
│ 阶段 1: 规划 (Planning)                  │
│ - 识别来源并读取内容                     │
│ - 判断是否需要 sources/ 摘要             │
│ - 提取关键信息                           │
│ - 查找现有页面                           │
│ - 输出规划表（页面列表 + 关系图）          │
│                                          │
│  [STOP] 等待用户确认 → 进入阶段 2         │
└─────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│ 阶段 2: 确认 (Confirmation)              │
│ - 用户审查规划表                         │
│ - 用户反馈修改意见                       │
│                                          │
│  如果用户有异议：                         │
│    └→ 回到阶段 1，修正规划，重新确认      │
│    (迭代循环，直到用户确认无问题)          │
│                                          │
│  用户确认无问题 → 进入阶段 3              │
└─────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│ 阶段 3: 执行 (Execution)                 │
│ - 创建/更新页面（含完整 relations）       │
│ - 修复现有页面的 relations               │
│ - 更新 index.md                         │
│ - 追加 log.md                           │
└─────────────────────────────────────────┘
```

### 详细步骤

#### 阶段 1: 规划（Planning）

**步骤 1.1: 识别来源类型并读取内容**

根据来源类型读取内容：
- **文件/代码**：读取用户指定的文件或代码文件
- **当前对话上下文**：从当前对话上下文中提取用户描述的概念、决策、业务理解等

**步骤 1.2: 判断是否需要 sources/ 摘要**

仅适用于**文件/代码**来源，创建 sources/ 摘要的条件：
- 文件 > 5000 行或 100 页
- 非代码文档（PDF/Word/会议记录/报告）
- 用户明确要求创建摘要

如果满足条件，计划在 `sources/xxx.md` 创建摘要页。
**对话来源跳过此步骤**。

**步骤 1.3: 提取关键信息**

从源内容中提取：
- **Overview**：是否涉及新的领域/模块，需要创建概览页？
- **Entity**：是否有新的或需要更新的领域实体？
- **Procedure**：是否有新的或需要更新的流程/操作？
- **Concept**：是否有新的或需要更新的抽象理念/模式？（可选，不要强行创建）
- **Synthesis**：是否触发需要综合分析？

**步骤 1.4: 查找现有页面（创建前必须执行）**

> **重要**：在建议创建任何新页面之前，必须先查找 wiki 中是否已存在相同或相似主题的页面。

**查找路径（按顺序执行）**：
1. 读取 `wiki/index.md`，了解当前 wiki 的完整索引结构，初步判断是否有相关页面
2. 按分类目录逐一扫描候选页面：
   - `wiki/overview/` - 检查是否有相关领域概览
   - `wiki/entities/` - 检查是否有相同或相关实体
   - `wiki/concepts/` - 检查是否有相同或相关概念
   - `wiki/procedures/` - 检查是否有相同或相关流程
   - `wiki/synthesis/` - 检查是否有相关综合分析
3. 对候选页面逐一读取内容，比较标题、摘要、核心内容与待创建内容的相似度

**判断匹配程度的标准**：

| 匹配程度 | 判断标准 | 操作建议 |
|----------|----------|----------|
| **高度匹配** | 现有页面主题与提取内容完全相同或高度重合（同一主题、同一实体、同一流程） | 建议**更新**现有页面 |
| **部分匹配** | 现有页面主题相关但覆盖范围不同，或仅涉及子主题 | 考虑**更新**现有页面并补充新内容，或创建新页面 |
| **无匹配** | wiki 中不存在相关主题的页面 | 可以建议**创建**新页面 |

**查找结果记录**：
- 对于建议创建的每个新页面，必须在规划表中添加"查找说明"，记录已检查的目录和候选页面
- 对于建议更新的每个已有页面，必须说明当前页面内容与待添加内容的关系

**步骤 1.5: 输出规划表**

LLM **必须**向用户展示规划表，分为四个部分：

```markdown
## Wiki Ingest 规划

我已从 [来源类型：文件名/当前对话] 中提取到以下关键信息，并从 wiki/index.md 开始逐步查找了现有页面。

### 待创建的页面（未在 wiki 中找到匹配）

| # | 页面路径 | 类型 | 内容摘要 | 提取理由 | 查找说明 |
|---|----------|------|----------|----------|----------|
| 1 | concepts/xxx.md | concept | 一句话描述该概念的核心内容 | 为什么从源内容中单独提取这条信息、它对 wiki 的知识积累有什么价值 | 已检查 concepts/ 目录，未发现匹配页面 |

### 待更新的页面（已存在于 wiki 中）

| # | 页面路径 | 类型 | 当前内容摘要 | 新增内容 | 提取理由 |
|---|----------|------|----------|----------|----------|
| 2 | entities/yyy.md | entity | 一句话描述该页面的现有内容 | 补充 xxx 细节 | 为什么补充这条信息、它对 wiki 的知识积累有什么价值 |

### 页面关系图

| 页面 A | ↔ | 页面 B | 关系描述 |
|----------|---|----------|----------|
| concepts/xxx.md | ↔ | entities/yyy.md | xxx 概念描述了 yyy 实体的核心机制 |
| overview/auth.md | ↔ | concepts/xxx.md | 概览页面包含 xxx 作为核心概念 |
| sources/architecture.md | ↔ | overview/auth.md | 源文件摘要作为概览的信息来源 |

### 需要修复的现有页面 relations

| 页面路径 | 当前 relations | 应新增关联 |
|----------|---------------|----------|
| entities/zzz.md | [overview/auth.md] | [overview/auth.md, concepts/xxx.md] |

---

请审查以上规划：
- 哪些页面需要创建/更新/跳过？（输入序号，如 1,2,3 或 全部）
- 关系图是否正确？
- 是否有需要调整的内容或关联关系？

如有修改意见，请告诉我，我会修正规划后重新提交确认。
如无问题，请确认进入执行阶段。
```

**[STOP] 等待用户确认后才能继续。**

#### 阶段 2: 确认（Confirmation）

用户审查阶段 1 的规划表并反馈。

**迭代循环规则**：
- **如果用户提出异议或要求修改**：
  - LLM 必须回到阶段 1，根据用户反馈修正规划
  - 重新输出修正后的规划表
  - 重新等待用户确认
  - **循环直到用户确认无问题**
- **如果用户确认无问题**：
  - 进入阶段 3 执行

**禁止行为**：
- 禁止：用户有异议后不回到阶段 1 修正就直接执行
- 禁止：自行假设用户的确认意图

#### 阶段 3: 执行（Execution）

用户确认规划无问题后，LLM 执行以下操作：

**步骤 3.1: 创建/更新 wiki 页面**

对于用户确认需要创建或更新的页面：
1. 按照 [wiki-structure.md](wiki-structure.md) 的模板格式
2. **创建页面时，front matter 中必须填写完整的 relations**
3. **更新页面时，必须同步更新 relations**
4. 文件大小建议详见 [wiki-structure.md](wiki-structure.md) - 文件大小控制

**步骤 3.2: 修复现有页面的 relations**

对于受本次 Ingest 影响的现有页面（其 relations 需要更新）：
1. 读取该页面内容
2. 更新 front matter 中的 relations 字段
3. 确保与新创建/更新页面的 relations 保持一致

```
对于本次 Ingest 创建或更新的每个页面 P：
  // 新页面 P 的 relations 已在步骤 3.1 中填写完整
  
  // 对于受影响的现有页面 X（X 的 relations 需要更新）：
  如果 P.relations 指向已有页面 X：
    将 P.path 添加到 X.relations（如不存在）
  
  // 对于新关联 P 的页面 N（N 在本次 Ingest 中被更新）：
  如果页面 N 的 relations 指向 P（N 在本次 Ingest 中被更新）：
    确保 N.path 已在 P.relations 中
```

**步骤 3.3: 更新 index.md**

按照 [log-index-guide.md](log-index-guide.md) 的规范更新全局索引。

**步骤 3.4: 追加 log.md**

按照 [log-index-guide.md](log-index-guide.md) 的格式追加活动日志。

**步骤 3.5: 展示执行结果**

创建/更新完成后，向用户展示：

```markdown
## 执行结果

我已执行以下操作：

### 创建的页面
- concepts/xxx.md
- entities/yyy.md

### 更新的页面
- overview/auth.md (补充 xxx)
- entities/zzz.md (修复 relations)

### 关系维护
- 新页面 relations 已填写完整
- 现有页面 relations 已修复

### 索引和日志
- index.md 已更新
- log.md 已追加记录

请确认内容是否正确。
```

**[STOP] 等待用户验证内容是否正确。**

### Ingest 示例 - 文件来源

```
用户：请读取 ARCHITECTURE.md 并更新 wiki

LLM 执行：
1. 来源类型：文件/代码 → 读取 ARCHITECTURE.md（1233 行）
2. 文件较大，计划在 sources/architecture.md 创建摘要
3. 提取关键信息
4. 【查找现有页面】从 wiki/index.md 开始，逐一检查 overview/ 和 concepts/ 目录，未发现匹配页面
5. 【阶段 1：规划】向用户展示规划表：
   
   ## Wiki Ingest 规划
   
   我已从 [文件/代码：ARCHITECTURE.md] 中提取到以下关键信息，并从 wiki/index.md 开始逐步查找了现有页面。
   
   ### 待创建的页面
   
   | # | 页面路径 | 类型 | 内容摘要 | 提取理由 | 查找说明 |
   |---|----------|------|----------|----------|----------|
   | 1 | sources/architecture.md | source | ARCHITECTURE.md 的摘要索引 | 文件较大(1233行)，创建摘要便于后续引用 | N/A |
   | 2 | overview/authentication.md | overview | 认证体系的整体设计和流程 | ARCHITECTURE.md 中描述了认证的整体架构，涉及多个子概念，值得创建 overview 页面作为导航入口 | 已检查 index.md 和 overview/ 目录，未发现覆盖认证主题的页面 |
   | 3 | concepts/agent-decision-loop.md | concept | Agent 通过循环调用决定下一步的机制 | 这是系统的核心工作机制，独立成页便于后续查询和关联其他概念 | 已检查 concepts/ 目录，未发现描述 Agent 决策循环的页面 |
   
   ### 待更新的页面
   
   (无)
   
   ### 页面关系图
   
   | 页面 A | ↔ | 页面 B | 关系描述 |
   |----------|---|----------|----------|
   | overview/authentication.md | ↔ | concepts/agent-decision-loop.md | 概览包含 Agent 决策循环作为核心概念 |
   | overview/authentication.md | ↔ | sources/architecture.md | 概览引用源文件摘要作为信息来源 |
   
   ### 需要修复的现有页面 relations
   
   (无)
   
   ---
   请审查以上规划。

6. 等待用户确认（如用户回复：全部创建，无问题）
7. 【阶段 3：执行】创建/更新页面，同时填写完整 relations
8. 更新 index.md 和 log.md
9. 向用户展示执行结果
```

   ### 待创建的页面

   | 页面 A | ↔ | 页面 B | 关系描述 |

8. 更新 index.md 和 log.md

## 工作流状态追踪

每个操作完成后，在 log.md 中追加记录：

```markdown
## [YYYY-MM-DD] ingest | 文件名

- 来源: 文件/代码 - path/to/file (行数/页数)
- 创建: wiki/xxx.md
- 更新: wiki/yyy.md, relations, index.md
```

## 询问用户模板

### 规划阶段

```
我已从 [来源] 中提取到关键信息，以下是 Wiki Ingest 规划：
[展示规划表]
请审查以上规划，确认哪些页面需要创建/更新/跳过？关系图是否正确？
```

### 确认阶段（迭代循环后）

```
我已根据你的反馈修正了规划，请重新审查：
[展示修正后的规划表]
```

### 执行完成后

```
我已执行了规划中的所有操作，请确认内容是否正确。
```

## 注意事项

- **三阶段流程必须严格执行** - 规划→确认→执行，不得跳过任何阶段
- **迭代循环** - 用户有异议时必须回到规划阶段，不得直接执行
- **relations 创建时必须填写** - 禁止"创建时留空后续填充"的模式
- **index.md 保持一致** - 每次页面变更后同步更新索引
